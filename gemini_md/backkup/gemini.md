# Code Design Principles
코드 작성 및 설계 기준.

- 분석 로직과 시각화 로직을 분리한다.
- 재사용 가능한 함수 단위로 작성한다.
- 매직 넘버 사용을 피하고 상수로 관리한다.
- 실험용 코드와 운영용 코드를 구분한다.
- 하나의 함수는 하나의 역할만 수행한다.
- 모듈화 할 수 있는 코드는 모듈화 하여 다른 코드에 활용할 수 있게 한다.

# Notebook Conventions
Jupyter Notebook 사용 시 규칙.

- 셀 실행 순서에 의존하지 않도록 작성
- 데이터 로딩 → 전처리 → 분석 → 시각화 → 결론 흐름 유지
- 중간 결과는 간단한 텍스트 요약 포함
- 그래프에는 반드시 제목, 축 라벨 포함



#  Analysis Report
- 설치된 라이브러리 , 분석방법, 시각화 자료, 사용된 모듈 등에 대한 정보는 report_analysis_파일명이 포함된 마크다운 파일에 상세하게 작성
- api 사용 시 사용된 토큰과 예상 비용(한화 기준)을 작성
- 진행과정에 발생된 에러에 대한 리포트도 작성
- 생성된 Report 자료는 reports 폴더에 정리


# Output

- 파일 생성 시 파일명에 끝에 '_현재 시간'을 입력
- 생성된 코드, 시각화 자료, 차트 등은 별도의 폴더에 저장
- 산출물은 제일 마지막에 알려줘



# File & Directory Structure
권장 디렉토리 구조.

text/
├─ data/
│  ├─ raw/             # 원본 데이터 (수정 금지)
│  └─ processed/       # 전처리된 데이터
├─ notebooks/          # 탐색적 분석 및 실험용 노트북
├─ src/                # 프로젝트 핵심 로직 (재사용 가능한 코드)
│  ├─ preprocessing/   # 데이터 로딩, 정제, 변환 관련 코드
│  ├─ analysis/        # 분석, 계산, 모델링 로직
│  └─ visualization/  # 시각화 함수 및 플롯 생성 코드
├─ output/             # 분석 결과 및 산출물
│  ├─ reports/         # 분석 리포트 (md, pdf 등)
│  ├─ figures/         # 이미지, 그래프, 차트
│  └─ artifacts/       # 모델, 중간 산출물, 기타 결과 파일
└─ README.md
